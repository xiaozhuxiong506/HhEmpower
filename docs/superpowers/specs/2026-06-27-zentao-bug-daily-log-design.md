# 禅道任务、Bug 与当日日志一体化设计

## 目标

扩展现有 TSB Browser MCP，使 Cursor 能一次采集禅道中“指派给我的任务”和“指派给我的 Bug”，按业务模块统一分析并分派执行。所有模块 Agent 完成后，只统计已经修改完成且验证通过的事项，并在每次流程结束时通过单选询问用户是否生成当日日志。

日志使用北京时间日期，按“模块 + 完成事项 + 耗时”输出。耗时以 0.5h 为步进，所有条目的总和必须严格等于 8.5h。

## 工具与兼容性

新增统一工具 `zentao_collect_assigned_work_items`，负责采集任务和 Bug。保留现有 `zentao_collect_assigned_tasks`，避免已有提示词或调用方式失效。

新增纯数据工具 `zentao_generate_daily_log`。该工具只接收已经完成并验证通过的工作项，不负责判断代码是否完成，也不会自行弹出确认框。Cursor 命令和技能负责在 Agent 全部返回后弹出单选，只有用户选择“生成今日日志”时才调用日志工具。

现有 `zentao_analyze_tasks` 保留；新增或扩展统一分析入口，使任务和 Bug 都使用相同的模块分类和 Agent 分派结构。

## 数据模型

任务和 Bug 统一表示为工作项：

```json
{
  "id": "4661",
  "kind": "task",
  "url": "http://hallzd.internal.tsb.com/zentao/task-view-4661.html",
  "title": "客户出货单生成应收单规则",
  "content": "",
  "remarks": "",
  "description": "",
  "module": "销售模块"
}
```

`kind` 只能是 `task` 或 `bug`。日志阶段的输入还需包含执行结果：

```json
{
  "id": "4661",
  "kind": "bug",
  "module": "销售模块",
  "title": "客户出货单生成应收单规则错误",
  "completionSummary": "修复客户出货单生成应收单规则，并补充校验",
  "completed": true,
  "verified": true
}
```

只有 `completed === true` 且 `verified === true` 的事项可以进入日志。失败、阻塞、仅分析、未修改完成或未验证的事项全部排除。

## 浏览器采集

统一采集工具打开禅道地盘页并复用持久化 Chrome Profile：

1. 进入“指派给我的任务”，读取 DTable 数据并跟随分页，逐条打开任务详情。
2. 进入“指派给我的 Bug”，读取 Bug DTable 数据并跟随分页，逐条打开 Bug 详情。
3. 任务链接使用 `task-view-{id}.html`，Bug 链接使用 `bug-view-{id}.html`。
4. 分别提取标题、内容、备注和描述；“暂无描述”、空备注和操作按钮文本不保留。
5. 合并为统一工作项列表，再按 ERP 模块分类并生成 `agentPlan`。

采集某一类失败时，结果需要明确返回任务或 Bug 的错误信息；另一类采集成功的数据仍然保留。登录失效时继续使用现有的人工登录提示。

## Agent 执行与确认

Cursor 命令读取 `groups` 和 `agentPlan`，按模块派发 `tsb-module-agent` 实例。每个 Agent 返回：完成事项、验证结果、失败或阻塞事项，以及适合日报使用的一句话完成摘要。

主 Agent 汇总全部结果后必须执行以下流程：

1. 过滤出完成且验证通过的事项。
2. 若没有符合条件的事项，说明没有可生成的日志，不弹出生成确认。
3. 若存在符合条件的事项，使用单选询问：
   - `生成今日日志（推荐）`
   - `暂不生成`
4. 选择“暂不生成”后结束，不输出日志。
5. 选择“生成今日日志”后调用 `zentao_generate_daily_log`。

此询问在每次任务/Bug 执行流程结束后都必须出现，不能根据历史选择自动跳过。

## 日志生成规则

日志标题为北京时间当天的 `YYYY-MM-DD 完成事项`。正文使用从 1 开始的有序列表，每项格式为：

```text
1.销售模块：修复客户出货单生成应收单规则问题 1.5h
```

生成规则如下：

- 描述必须清晰包含业务模块和完成结果，不直接复制冗长的禅道原文。
- 不显示禅道任务号或 Bug 编号。
- 相同模块、相同功能主题的细碎事项先归纳合并。
- 日志最多 17 项；超过时继续按模块和功能主题合并，保证每项至少可分配 0.5h。
- 根据事项复杂度分配半小时单位，总共分配 17 个半小时单位，即 8.5h。
- 每项耗时必须是 0.5h 的整数倍，且所有项相加严格等于 8.5h。
- 只有一个完成事项时，该事项记为 8.5h。
- 同时包含任务和 Bug 时总结为 `总结：日常任务及Bug开发`；仅有任务时为 `总结：日常任务开发`；仅有 Bug 时为 `总结：日常Bug修复`。

## 输出示例

```text
2026-06-27 完成事项
1.销售模块：修复客户出货单生成应收单规则问题 1.5h
2.财务模块：完成厂商付款审批详情页面 2h
3.财务模块：补充客户预收与收款流程审批及红冲功能 2.5h
4.公共基础模块：优化查货剩余数量过滤规则 1h
5.公共基础模块：完成流程重新提交后的页面定位与弹窗联动 1.5h
总结：日常任务及Bug开发
```

示例总耗时为 8.5h。

## 测试与验收

自动化测试覆盖：

- 任务与 Bug DTable 链接提取、分页和详情解析。
- 统一工作项分类及 Agent 计划。
- 只保留完成且验证通过的日志输入。
- 相似事项合并以及超过 17 项时的压缩。
- 北京时间日期。
- 每项耗时为 0.5h 的整数倍，合计严格为 8.5h。
- MCP 工具列表和调用结果。
- Cursor 命令与技能包含结束后的强制单选步骤。

最后使用真实禅道登录态验证任务和 Bug 均能采集，且旧任务工具仍可正常使用。

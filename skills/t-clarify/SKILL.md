# t-clarify — 需求澄清（硬卡停关卡）

> **定位**：横向前置 Skill，横跨 `/t-start`、`/t-new-module`、`/t-review`、`/t-zentao-collect`。
> 不是 agent（不写码），是教主 Agent **何时问、问什么、怎么问**。
> 与 `t-start` 的 AskUserQuestion 不冲突：那个处理「范围不清=改哪个文件」，本 Skill 处理「意图不清=想达到什么效果」。

## 何时触发（主 Agent 必做扫描）

在执行任何 command 的实质动作（读 workflow / 出蓝图 / 派 agent / 取 diff）**之前**，
先扫描用户原始需求文本，命中下表任一信号词即触发本 Skill。

**例外（不触发）**：信号词在上下文里有明确指代时跳过。例：用户刚说"看下采购订单列表"，紧跟"优化下它的查询"——"它"有指代，不卡停。

## 信号词 → 澄清方向表

| 信号词 | 模糊类型 | 必问的澄清问题 | 候选项怎么来 |
|--------|---------|---------------|------------|
| 那个 / 那个页面 / 那个功能 / 那个东西 | **指代不明** | "你说的『那个』是指哪个模块/页面？" | glob 列 `src/views/${前缀}*/` 下候选，让用户单选 |
| 优化下 / 优化一下 / 改善 / 调整下 | **目标不明** | "你想优化的是哪个维度？" | 性能 / 可读性 / 交互体验 / 代码规范（四选一） |
| 按上次 / 按之前 / 和原来一样 / 老样子 | **参照不明** | "『上次』是指哪一次改动？" | `git log --oneline -10` 取最近提交让用户指认 |
| 加个查询 / 加个功能 / 加个东西 | **范围不明** | 三连问：加在哪（列表/详情/弹窗）+ 查什么（哪些字段）+ 触发方式（按钮/自动） | 列表/详情/弹窗三选一 + 字段多选 |
| 随便 / 差不多 / 看着办 / 都行 | **标准不明** | "我给两个具体方案你挑，还是按我的推荐来？" | 方案A / 方案B / AI推荐（三选一） |
| 处理一下 / 搞一下 / 弄一下 / 搞定它 | **动作不明** | "你想让我做的是？" | 修复bug / 新增功能 / 重构代码（三选一） |
| 像XX那样 / 和XX一样 / 参照XX | **类比不明** | "XX 的哪些特征你要？" | 列出 XX 的 3-5 个特征让用户勾选 |
| 快一点 / 慢 / 卡 | **现象不明** | "慢/卡出现在什么操作时？" | 首次加载 / 翻页 / 弹窗打开 / 保存（多选） |
| 不对 / 有问题 / 坏了 / 报错了 | **故障不明** | "具体什么现象？" | 截图 / 报错信息 / 复现步骤（让用户提供） |

## 澄清三问（统一产出格式）

命中信号词后，主 Agent 用 `AskUserQuestion` 一次性发起 **1–3 个单选题**（按信号词类型决定问几个），格式严格如下：

```
问题 1（定位类）：你说的『XX』具体是指？
  ○ 候选 A   ○ 候选 B   ○ 候选 C

问题 2（目标类，可选）：你期望的效果是？
  ○ 方案 A   ○ 方案 B

问题 3（验收类，可选）：怎么算做完？
  ○ 标准 A   ○ 标准 B
```

**提问纪律**（避免烦人）：
- ✅ 只问信号词对应的必问项，不追问无关细节
- ✅ 候选项必须具体（不能是"其他"敷衍），最多 4 个
- ✅ 第一选项放 AI 最可能的解读（标注"(推荐)"）
- ✅ 能从代码/git 里推断的候选，直接列出来，别让用户打字
- ❌ 不问开放题（"你想怎么做？"）
- ❌ 不超过 3 个问题
- ❌ 不重复问已在上文确认过的

## 硬卡停规则（不可绕过）

触发本 Skill 后，**在用户回答澄清三问之前**：

- ❌ 不得 Read 任何 `t-workflow` / `t-list` 等下游 Skill
- ❌ 不得派任何 subagent（`t-scout` / `t-api` 等）
- ❌ 不得出蓝图、不得写码、不得取 diff
- ❌ 不得说"我先按理解做，不对再改"——这正是要避免的反模式

唯一允许的动作：发起 `AskUserQuestion` 澄清三问。

## 放行条件（答完才走）

用户回答后，主 Agent 用一句话**复述确认**：

> "明白，你是要【模块 X】做【动作 Y】，做完的标准是【Z】。对的话我开始，不对请纠正。"

用户确认（或 3 秒内无异议即视为认可）后，把澄清结论作为 **context 头** 写入任务上下文：

```
【已澄清需求】模块=t-purchase-order-list | 动作=新增多字段查询 | 验收=支持按供应商+日期+状态组合筛选
```

随后放行给对应 command 的正常流程。

## 退出与移交

澄清完成后，本 Skill 退出，**不直接派 agent**。由后续 command（`/t-start` 等）按其既定流程接手。本 Skill 只负责"把需求问清楚"，不负责"执行需求"。

## 与现有流程的接入点

| Command | 插入位置 | 说明 |
|---------|---------|------|
| `/t-start` | 第 0 步（探测项目）之后、读 workflow 之前 | 最常用入口 |
| `/t-new-module` | 探测项目之后、出蓝图之前 | 新建模块也要先问清"做什么页面、哪些字段" |
| `/t-review` | 取 diff 之前 | 用户说"审查下那个改动"时先定位是哪个改动 |
| `/t-zentao-collect` | 禅道任务分组之后、派 `t-mod` 之前 | 禅道任务标题常含糊，分组后让用户确认 |

## 信号词扫描优先级

若需求里同时命中多个信号词，按此顺序逐个澄清（一次 AskUserQuestion 可合并问，但不超过 4 题）：

1. 指代不明（那个）—— 最高优先级，不定对象后面全白问
2. 故障不明（报错了） / 现象不明（卡）
3. 动作不明（处理一下）
4. 范围不明（加个查询） / 目标不明（优化下）
5. 参照不明（按上次） / 类比不明（像XX） / 标准不明（随便）
